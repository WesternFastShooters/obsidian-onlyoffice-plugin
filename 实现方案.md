# Obsidian OnlyOffice 插件实现方案

## 概述

在 Obsidian 内 **编辑与预览** Office 文档（Word / Excel / PPT），默认进入编辑态。所有文档处理在本地完成，保存时直接覆盖 Vault 原文件。

参考项目：[sweetwisdom/onlyoffice-web-local](https://github.com/sweetwisdom/onlyoffice-web-local)（纯前端、WASM 转换、OnlyOffice WebSDK、无需后端服务）

---

## 1. 确定的需求与决策

| 项目 | 决策 |
|---|---|
| 许可证 | 个人使用，不公开分发，不考虑 AGPL 合规问题 |
| 平台 | **仅桌面端**（Electron）。移动端不支持（见下方说明） |
| 文件格式 | 优先 `.docx/.xlsx/.pptx`；旧格式 `.doc/.xls/.ppt` 打开时提示另存为对应 OpenXML 格式后编辑 |
| 默认模式 | 打开即进入**编辑态**（可通过命令切换为只读） |
| 保存策略 | **覆盖原文件**，不做另存为、不做自动备份（备份由 Obsidian/同步工具自身管理） |
| 冲突处理 | 不做冲突检测（文件同步一致性由 Obsidian 客户端/第三方同步服务保障） |
| 联网策略 | 默认允许网络请求；但核心编辑/预览功能必须**离线可用**（所有 WebSDK + WASM 资源本地打包） |
| 导出 | 支持导出为不同格式（源码已确认支持：PDF、DOCX、DOC、ODT、RTF、TXT、XLSX、XLS、ODS、CSV、PPTX、PPT、ODP 等） |
| 命令面板 | 注册 Obsidian 命令：保存、另存为、导出 PDF、切换只读/编辑 |
| 打开速度 | 目标 1s 内打开；通过预热/缓存实现二次秒开 |

### 移动端不支持的原因

- OnlyOffice 开源版 WebSDK **不包含移动端编辑器**（移动端编辑仅商业版 Enterprise/Developer 提供）
- x2t-wasm（55 MB）在移动端 WebView 内存/性能受限
- `onlyoffice-web-local` 项目本身无任何移动端适配代码
- **策略**：检测 `Platform.isMobile`，显示"Office 编辑仅支持桌面端"提示

## 2. 源码深度分析（基于本地 clone 的完整源码）

### 2.1 资源体积（实际测量）

| 资源 | 大小 | 用途 | 是否必须 |
|---|---|---|---|
| `wasm/x2t/x2t.wasm` | 55 MB | 文档格式转换核心 | 必须 |
| `wasm/x2t/x2t.js` | 188 KB | WASM 加载器 | 必须 |
| `sdkjs/word/` | 8 MB | Word 编辑器 SDK | 按需（打开 docx 时） |
| `sdkjs/cell/` | 9 MB | Excel 编辑器 SDK | 按需（打开 xlsx 时） |
| `sdkjs/slide/` | 9.6 MB | PPT 编辑器 SDK | 按需（打开 pptx 时） |
| `sdkjs/common/` | 5.5 MB | 共享库（含字体 WASM） | 必须 |
| `web-apps/` | 11 MB | 编辑器 UI（含 documenteditor/spreadsheeteditor/presentationeditor） | 必须 |
| `fonts/` | 80 MB（16 个文件） | 字体文件 | **仅 PDF 导出需要** |
| **合计** | **~178 MB** | — | 不含 fonts 约 98 MB |

**优化决策**：fonts 目录（80 MB）**不随插件分发**，仅在用户首次使用 PDF 导出时提示下载或手动放置。这样插件核心资源降至 ~98 MB。

### 2.2 编辑器实际加载机制（关键发现，与原方案有出入）

通过阅读 `api.js` 源码，发现了一个对架构有根本影响的事实：

**`DocsAPI.DocEditor` 内部一定会创建 iframe。** 它通过检查自身 `<script>` 标签的 `src` 属性来推导编辑器资源的基础路径：

```javascript
// api.js 内部逻辑（简化）
function getBaseUrl() {
  var scripts = document.getElementsByTagName('script');
  for (var i = scripts.length - 1; i >= 0; i--) {
    var match = scripts[i].src.match(/(.*)api\/documents\/api.js/i);
    if (match) return match[1]; // 例如 "file:///path/to/web-apps/apps/"
  }
}
// 然后拼接 iframe src:
// baseUrl + "documenteditor/main/index.html?lang=zh&..."
// baseUrl + "spreadsheeteditor/main/index.html?lang=zh&..."
// baseUrl + "presentationeditor/main/index.html?lang=zh&..."
```

**这意味着**：
1. 无论我们选择哪种方案，编辑器内部都有一层 iframe
2. `api.js` 所在的目录结构**必须完整保留**（`web-apps/apps/api/documents/api.js` 相对于 `web-apps/apps/documenteditor/` 等的位置）
3. 编辑器 iframe 内部再通过相对路径加载 `sdkjs/` 等资源

### 2.3 x2t 转换模块的全局变量

`x2t.js` 加载后会在 `window.Module` 上挂载 Emscripten 模块。初始化时通过 `onRuntimeInitialized` 回调通知就绪，然后使用 `Module.FS`（虚拟文件系统）和 `Module.ccall` 执行格式转换。

全局变量冲突点：
- `window.Module`（x2t WASM）
- `window.DocsAPI`（编辑器 API）

### 2.4 保存流程的真实数据结构

```typescript
// onSave 事件结构
interface SaveEvent {
  data: {
    data: string;            // bin 数据（编辑器内部格式）
    option: {
      outputformat: number;  // 格式码，如 65=DOCX, 257=XLSX, 129=PPTX
    }
  }
}

// 通过 c_oAscFileType2 映射：65 → "DOCX", 257 → "XLSX" 等
// 然后调用 convertBinToDocumentAndDownload(data.data, fileName, "DOCX")
// 返回 { fileName: string, data: Uint8Array }
```

### 2.5 图片粘贴处理（writeFile 事件）

当用户在编辑器中粘贴图片时，编辑器会触发 `writeFile` 事件，携带图片的 `Uint8Array` 数据和文件名。处理流程：
1. 将图片数据创建为 `Blob` → `URL.createObjectURL` 生成 blob URL
2. 通过 `asc_setImageUrls` 命令将 URL 映射注册到编辑器
3. 通过 `asc_writeFileCallback` 通知编辑器图片已就绪

**在 Obsidian 插件中**：可以将粘贴的图片保存到 Vault 中（与 Obsidian 的附件管理策略一致），再把 Vault 路径转为可访问的 URL 传回编辑器。

### 2.6 空文档模板

`empty_bin.ts` 导出 `g_sEmpty_bin` 对象，包含 `.docx`、`.xlsx`、`.pptx` 的空文档 bin 数据（base64 编码）。可用于实现"新建 Office 文档"功能。

### 2.7 不需要 Vue/Vite

`onlyoffice-web-local` 使用 Vue 3 + Vite + Element-Plus + Pinia 构建 UI，但这些全是该项目自己的 Web UI 框架。对于 Obsidian 插件，我们**完全不需要**这些，只需提取：
- `x2t.ts`（转换逻辑）→ 直接复制改造为 TypeScript 模块
- `empty_bin.ts`（空模板）→ 直接复制
- `public/` 下的静态资源 → 直接复制，保持目录结构
- `api.js` → 通过 `<script>` 动态加载

## 3. 总体架构（修正后）

### 方案重新评估

原方案提出了"方案 A：直接嵌入"和"方案 B：iframe 隔离"的选择。基于源码分析的修正：

| | 方案 A：在 Obsidian 主窗口加载脚本 | 方案 B：外层 iframe 隔离 |
|---|---|---|
| 原理 | 在 Obsidian 窗口的 `<head>` 中注入 `x2t.js` 和 `api.js`，DocEditor 内部创建编辑器 iframe（不可避免） | 创建一个外层 iframe 加载一个轻量 HTML，其中再加载 `api.js`，DocEditor 在这个 iframe 内创建编辑器 iframe |
| iframe 层数 | 1 层（DocEditor 自己创建的） | 2 层（我们的 + DocEditor 创建的） |
| 全局变量影响 | `window.Module` 和 `window.DocsAPI` 挂在 Obsidian 主窗口上，有冲突风险 | 全局变量隔离在外层 iframe 中，不影响 Obsidian |
| 脚本路径 | 需要 `file://` 绝对路径加载 | 外层 iframe 的 src 用 `file://` 加载本地 HTML，内部用相对路径 |
| 通信复杂度 | 直接调用 `sendCommand`，简单 | 需要 `postMessage` 穿透外层 iframe，稍复杂 |
| 稳定性 | 低（全局污染风险） | 高（完全隔离） |

**修正决策：推荐方案 B（iframe 隔离）。**

理由：
1. `window.Module` 是 Emscripten 的标准全局变量名，Obsidian 或其他插件也可能使用 WASM，冲突概率不低
2. 方案 B 只需要写一个轻量 `editor.html`（几十行），复杂度增加很小
3. 通信协议只需要 `open`/`save`/`export` 几个消息，`postMessage` 开销可以忽略
4. 编辑器内部本来就用 iframe + postMessage，多一层不会有感知延迟

### 3.1 架构图

```
Obsidian 主窗口
│
├─ OfficeFileView (自定义 FileView)
│   │
│   ├─ 外层 iframe (src = "file://.../assets/onlyoffice/editor.html")
│   │   │
│   │   ├─ editor.html 加载 x2t.js + api.js
│   │   ├─ editor.html 内的 JS 处理 postMessage 通信
│   │   │
│   │   └─ DocsAPI.DocEditor 内部 iframe (自动创建)
│   │       └─ documenteditor/main/index.html (或 spreadsheet/presentation)
│   │           └─ 加载 sdkjs/word/sdk-all-min.js 等
│   │
│   └─ 状态栏 (加载中 / 已保存 / 错误)
│
├─ VaultIO (读写 Vault 二进制文件)
└─ 命令注册 (保存 / 另存为 / 导出 PDF / 切换模式)
```

### 3.2 `editor.html`（外层 iframe 页面，约 80 行）

这是一个**我们自己编写的**轻量 HTML 页面，放在 `assets/onlyoffice/editor.html`，职责：
- 加载 `x2t.js` 和 `api.js`
- 监听 `postMessage`，接收插件侧发来的 `open` / `export` 指令
- 调用 x2t 转换 + DocEditor 创建
- 将保存结果通过 `postMessage` 回传插件侧

### 3.3 模块划分

```
src/
├── main.ts                  # 插件入口（注册扩展名、命令）
├── OfficeFileView.ts        # 自定义 FileView（承载外层 iframe + 状态栏）
├── EditorBridge.ts          # 封装与 editor.html iframe 的 postMessage 通信

├── commands.ts              # Obsidian 命令注册
└── types.ts                 # 通信协议类型定义

├── editor.html              # 我们编写的外层 iframe 页面
├── editor.js                # editor.html 的逻辑（x2t 调用 + DocEditor 管理 + 消息处理）
├── x2t-converter.js         # 从 x2t.ts 提取、改造的转换模块
├── empty-bin.js             # 空文档模板
├── web-apps/                # 保持原目录结构
│   └── apps/
│       ├── api/documents/api.js
│       ├── documenteditor/
│       ├── spreadsheeteditor/
```

### 3.4 通信协议

插件侧（`EditorBridge`）↔ 外层 iframe（`editor.html`）之间用 `postMessage` 通信。

```typescript
// 插件 → iframe
type PluginMessage =
  | { type: 'oo:open'; requestId: string; payload: { fileName: string; fileData: ArrayBuffer; readonly: boolean } }
  | { type: 'oo:save'; requestId: string }
  | { type: 'oo:export'; requestId: string; payload: { targetExt: string } }

// iframe → 插件
type EditorMessage =
  | { type: 'oo:ready' }
  | { type: 'oo:opened'; requestId: string }
  | { type: 'oo:saved'; requestId: string; payload: { fileData: ArrayBuffer; fileName: string } }
  | { type: 'oo:exported'; requestId: string; payload: { fileData: ArrayBuffer; fileName: string } }
  | { type: 'oo:error'; requestId: string; payload: { code: string; message: string } }
  | { type: 'oo:image-paste'; payload: { imageData: ArrayBuffer; fileName: string } }
```

数据传输使用 `ArrayBuffer` + `Transferable`，避免拷贝和 Base64 膨胀。

## 4. 文件生命周期

### 4.1 打开文件

```
用户点击 .docx 文件
  │
  ▼
Obsidian 触发 OfficeFileView.onLoadFile(file)
  │
  ├─ VaultIO.readBinary(file) → ArrayBuffer
  │
  ├─ 创建/复用外层 iframe（src = file://.../editor.html）
  │
  ├─ 等待 iframe 发送 oo:ready
  │
  └─ 发送 oo:open { fileName, fileData: ArrayBuffer, readonly: false }
       │
       ▼ (iframe 内部)
       ├─ 确保 x2t 已初始化（单例，幂等）
       ├─ new File(fileData, fileName)
       ├─ convertDocument(file) → { bin, media, type }
       ├─ new DocsAPI.DocEditor('editor-container', config)
       └─ onAppReady → asc_setImageUrls(media) + asc_openDocument(bin)
            │
            ▼
       iframe 发送 oo:opened
```

### 4.2 打开旧格式（.doc / .xls / .ppt）

```
用户点击 .doc 文件
  │
  ▼
OfficeFileView.onLoadFile(file)
  │
  ├─ 检测到旧格式扩展名
  ├─ 弹出 Obsidian 模态框提示：
  │  "该文件为旧格式(.doc)，将转换为 .docx 后编辑。是否继续？"
  │
  ├─ 用户确认 → 正常流程，打开编辑器
  │  └─ 保存时写为 .docx（同目录，文件名不变仅换扩展名）
  │
  └─ 用户取消 → 不打开
```

### 4.3 保存（覆盖原文件）

```
用户按 Ctrl+S / 执行"保存"命令
  │
  ▼
OfficeFileView.save()
  │
  ├─ 发送 oo:save 给 iframe
  │    │
  │    ▼ (iframe 内部)
  │    ├─ 编辑器 onSave 事件 → 获得 bin 数据 + outputformat 格式码
  │    ├─ c_oAscFileType2[outputformat] → 格式字符串（如 "DOCX"）
  │    ├─ convertBinToDocument(bin, fileName, targetExt)
  │    │  → Uint8Array（不弹出下载框，仅返回数据）
  │    ├─ asc_onSaveCallback({ err_code: 0 }) 通知编辑器保存完成
  │    └─ postMessage oo:saved { fileData: ArrayBuffer, fileName }
  │
  ├─ 收到 oo:saved
  ├─ VaultIO.writeBinary(file, arrayBuffer)
  │  → this.app.vault.modifyBinary(file, arrayBuffer)
  │
  └─ 更新状态栏："已保存"
```

### 4.4 另存为 / 导出

```
用户执行"导出为 PDF"命令
  │
  ▼
OfficeFileView.exportAs('PDF')
  │
  ├─ 发送 oo:export { targetExt: 'PDF' } 给 iframe
  │    │
  │    ▼ (iframe 内部)
  │    ├─ 获取当前 bin → convertBinToDocument(bin, fileName, 'PDF')
  │    └─ postMessage oo:exported { fileData, fileName: "xxx.pdf" }
  │
  ├─ 收到 oo:exported
  ├─ 计算新路径：同目录 / 同名.pdf
  └─ VaultIO.createBinary(newPath, arrayBuffer)
```

### 4.5 图片粘贴

```
用户在编辑器中粘贴图片
  │
  ▼ (iframe 内部)
  编辑器触发 writeFile 事件
  │
  ├─ 方案 1（简单）：直接 Blob URL 处理（与原项目一致）
  │  └─ 图片只在编辑器会话中有效，保存时嵌入文档
  │
  └─ 方案 2（增强，M3 做）：
     ├─ postMessage oo:image-paste { imageData, fileName } 给插件
     ├─ 插件保存到 Vault 附件目录
     ├─ 返回 Vault 文件的 file:// URL
     └─ 编辑器使用该 URL 引用图片
```

### 6.1 从源码提取资源（不需要构建 Vue 项目）

### 6.2 我们自己编写的文件

从 `x2t.ts` 提取转换逻辑，改造为独立模块（去掉 Vue 依赖、修改脚本路径为相对路径、去掉下载逻辑）：

- `assets/onlyoffice/editor.html` — 外层 iframe 页面

- `assets/onlyoffice/x2t-converter.js` — 从 x2t.ts 改造的转换模块

### 6.3 脚本加载路径策略

`editor.html` 用 `file://` 协议从插件目录加载，内部使用**相对路径**引用所有资源：

```html
<!-- editor.html -->
<script src="./wasm/x2t/x2t.js"></script>
<script src="./web-apps/apps/api/documents/api.js"></script>
<script src="./x2t-converter.js"></script>
<script src="./editor.js"></script>
```

由于 `api.js` 通过查找自身 `<script>` 标签的 `src` 来推导编辑器路径，**只要目录结构保持完整**，它就能正确找到 `documenteditor/main/index.html` 等编辑器页面。

插件侧获取 `editor.html` 路径：

```typescript
const basePath = (this.app.vault.adapter as FileSystemAdapter).getBasePath();
const pluginDir = `${basePath}/.obsidian/plugins/obsidian-onlyoffice`;
const editorUrl = `file://${pluginDir}/assets/onlyoffice/editor.html`;
```

## 7. 性能优化（目标：1s 内打开）

### 7.1 首次加载

首次打开 Office 文件时需加载 x2t WASM（55 MB）+ WebSDK，预计 3-5 秒。优化手段：

- **加载骨架屏**：显示"正在初始化编辑器..."的 loading 状态
- **异步并行**：x2t 脚本和 api.js 并行加载
- **x2t 和 api.js 分离预热**：x2t WASM 编译是最耗时的步骤，可以在插件启动时就开始

### 7.2 二次打开（秒开）

- **iframe 复用**：切换 Office 文件时不销毁外层 iframe，只发新的 `oo:open` 指令
- **x2t 单例**：x2t 模块（`X2TConverter`）在 iframe 内是单例，只初始化一次
- **插件启动预热**：通过 `requestIdleCallback` 在 Obsidian 空闲时创建隐藏 iframe 并预加载 x2t + api.js，这样用户打开文件时脚本已就绪

### 7.3 iframe 复用策略

```typescript
// 插件持有一个 iframe 引用
private editorIframe: HTMLIFrameElement | null = null;

// 首次打开时创建 iframe 并缓存
// 后续打开时复用 iframe，只发 oo:open 消息
// 切换文件类型（word → excel）时可能需要重建 DocEditor（因为编辑器类型不同）
```

### 7.4 大文件处理

- `postMessage` 传输使用 `Transferable`（零拷贝）：
  ```typescript
  iframe.contentWindow.postMessage(msg, '*', [msg.payload.fileData]);
  ```
- 转换操作在 iframe 内执行（独立于 Obsidian UI 线程，不会卡 Obsidian 界面）

## 8. 风险清单

| 风险 | 影响 | 对策 |
|---|---|---|
| `file://` iframe 加载被 Electron CSP 阻止 | iframe 无法加载 editor.html | POC 首先验证；备选：用 `iframe.srcdoc` 注入 HTML，资源改为绝对 `file://` URL |
| `api.js` 路径推导失败 | DocEditor 无法找到编辑器 HTML | 确保 api.js 的 script.src 是完整 `file://` 路径且目录结构完整 |
| 编辑器 iframe（DocEditor 内部创建的）加载 sdkjs 资源失败 | 编辑器空白或报错 | POC 验证；备选：修改 api.js 内的路径逻辑 |
| WASM 资源体积大（~98 MB 不含 fonts） | 插件包过大 | 个人使用可接受；可选：把资源放在 Vault 外单独目录 |
| 旧格式转换不完整 | `.doc/.xls/.ppt` 内容丢失或样式错乱 | 提示用户另存为 OpenXML；POC 中验证转换质量 |
| OnlyOffice 编辑器在 Electron 中的渲染兼容性 | 渲染异常 | POC 第一优先级验证 |
